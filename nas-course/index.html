What This Machine Does

Imagine you could design a brain for a tiny robot — not by guessing, but by measuring exactly what fits inside its skull. That’s what this project does for AI.

The Problem: One Size Does NOT Fit All

A neural network that runs fine on a powerful cloud server will choke on a tiny edge device. It’s like forcing a size-14 shoe onto a size-8 foot.

☁️

Cloud GPU Server

80 GB RAM • 300+ TFLOPS • No power limit • Runs any model easily — but costs $10,000+

🤖

Edge Device (Jetson)

8 GB RAM • 17 TFLOPS • 15W power limit • Must run AI locally, in real time, on a $199 board

🪡
The Tailor Metaphor

Think of a master tailor. Instead of making one suit that sort-of fits everyone, the tailor measures the exact body (the hardware), consults a fabric catalog (the building blocks), and creates the perfect pattern (architecture) that fits perfectly — not too big (slow), not too small (inaccurate).

Why not just use a smaller model?

You could shrink a big model, but that’s like hemming a size-14 suit to fit a size-8 body — it’ll look bad. This project designs the suit from scratch, tailored to the exact measurements of the Jetson.

The Four-Piece Strategy

This project uses four interlocking strategies to find the perfect architecture. Each one handles a different part of the tailoring process.

1
Latency LUT — The Measuring Tape

Measures every building block on the real Jetson device. Instead of guessing speed from FLOPs, it records actual latency in milliseconds.

2
OFA Supernet — The Fabric Catalog

A single large supernet pre-trained so that any subnet inside it can be extracted and immediately works well — like a walk-in closet full of ready-to-wear outfits.

3
Bayesian Optimization — The Expert Eye

Bayesian Optimization learns from each attempt to pick the next architecture to try — far smarter than random guessing.

4
Net2Net — The Alterations Expert

Recycles trained weights from a similar previous architecture so each new evaluation starts from a good place (warm-start) instead of from random noise.

Each round, BO gets smarter about which architectures are worth trying. After enough rounds, it converges on the Pareto frontier — the set of best possible tradeoffs between accuracy and speed.

The Objective Function: How Architectures Are Scored

Every candidate architecture gets a single score. The search maximizes this formula — balancing accuracy against hardware constraints.

CODE 

J(α) = accuracy(α) − λ · latency_ms(α) − μ · max(0, mem(α) − mem_budget)²
PLAIN ENGLISH

J(α) = the score of architecture α

accuracy(α) = how accurate the model is (higher is better)

− λ · latency_ms(α) = MINUS a penalty for being slow

− μ · max(0, mem − budget)² = MINUS a big penalty if it uses too much memory

🎯
Why “squared” for memory?

The squared penalty means going slightly over budget is a small hit, but going way over budget is devastating. It’s a hard wall: stay within 8 GB or the score tanks.

You have two architectures: A gets 82% accuracy at 15ms latency, B gets 79% accuracy at 8ms latency. If your robot needs to react within 10ms, which would you pick?

Why should I care?

Understanding this system teaches you how real AI deployment works. It is not just about accuracy — it is about running fast on real hardware within real constraints. Every self-driving car, every drone, every smart camera faces this exact problem. You can fine-tune a model all day, but if it does not fit the device, it is useless.

The Measurement Machine

How the LUT pipeline benchmarks every building block on real hardware — so the search engine never has to guess.

How fast is a 5×5 depthwise convolution with 96 channels at 28×28 on a Jetson Orin Nano?

You could guess. Or you could actually measure it. This module is about the system that actually measures it.

🔬

5×5 Depthwise Conv

96 channels, 28×28 resolution

FLOPs: 3.6M

⚡ 0.04 ms on Jetson

🔬

3×3 Regular Conv

96→96 channels, 28×28 resolution

FLOPs: 3.5M

🐢 0.08 ms on Jetson

Nearly the same FLOPs — but the depthwise conv is 2× faster because it reads far less memory. FLOPs lie. Measurement doesn't.

🧠
The Memory-Bandwidth Effect

This mismatch between FLOPs and actual speed is called the memory-bandwidth effect. Most neural network operations on GPUs are limited by how fast they can read data from memory, not by how fast they can do math.

The Block Registry: MBConv

Think of a car testing facility: before an F1 team races, they test every component in a wind tunnel and record the results. The inverted bottleneck block (MBConv) is the most-tested component in our wind tunnel.

CODE 

class MBConv(nn.Module):
    def __init__(self, in_c: int, out_c: int, kernel: int, stride: int,
                 expand: int, se: bool):
        super().__init__()
        mid = in_c * expand
        layers = []
        if expand != 1:
            layers += [nn.Conv2d(in_c, mid, 1, 1, 0, bias=False),
                       nn.BatchNorm2d(mid), nn.ReLU6(inplace=True)]
        pad = kernel // 2
        layers += [nn.Conv2d(mid, mid, kernel, stride, pad, groups=mid, bias=False),
                   nn.BatchNorm2d(mid), nn.ReLU6(inplace=True)]
        if se:
            layers.append(SEBlock(mid))
        layers += [nn.Conv2d(mid, out_c, 1, 1, 0, bias=False),
                   nn.BatchNorm2d(out_c)]
        self.op = nn.Sequential(*layers)
        self.residual = (stride == 1 and in_c == out_c)

    def forward(self, x):
        y = self.op(x)
        return x + y if self.residual else y
PLAIN ENGLISH

Create a mobile inverted bottleneck block — the workhorse of efficient neural networks.

The expanded width = input channels × expand ratio (e.g., 64 × 6 = 384).

If we're expanding (not 1:1), add a 1×1 conv to widen the channels + BatchNorm + ReLU6 activation.

Add a depthwise conv — each channel gets its own filter, very efficient on GPUs.

Optionally add Squeeze-and-Excitation (SE): a tiny attention mechanism that learns which channels matter.

Project back down to the output width with a 1×1 conv.

If input and output shapes match, add a residual connection — helps gradients flow during training.

Forward pass: run through all layers. If residual, add the original input back in.

The Wind Tunnel Analogy

The row_key Trick

Every wind tunnel test needs a serial number. The LUT uses a SHA-1 hash as a fingerprint for each measurement.

CODE 

def row_key(block: str, cfg: dict, input_shape) -> str:
    """Stable ID for a LUT row — used for resumability and engine filenames."""
    payload = json.dumps({"b": block, "c": cfg, "s": list(input_shape)},
                         sort_keys=True)
    return hashlib.sha1(payload.encode()).hexdigest()[:16]
PLAIN ENGLISH

Given a block type, its configuration, and the input tensor shape...

Pack them into a single JSON string with sorted keys (so the same config always produces the same string).

Hash it with SHA-1 and take the first 16 characters — that's your unique row_key.

Source: catalog/sweep.py lines 8–12

🔑
Content-Addressing

This pattern — hashing inputs to create a stable, unique ID — is called content-addressing. Git uses the same idea: every file gets an ID based on its contents, not its name. If two different runs produce the same block config, they get the same row_key and the LUT only stores it once.

1
Same input → Same hash

MBConv with kernel=3, expand=6, 96 channels always produces the same row_key.

2
Crash? No problem.

On restart, the sweep reads existing keys from lut.jsonl and skips any already measured. This is idempotent design.

3
16 chars is enough

With 16 hex characters there are 264 possible keys — collision is astronomically unlikely for 2,710 rows.

The Export Step: PyTorch → ONNX

The laptop builds each block in PyTorch. Then it needs to hand off a blueprint to the Jetson. That blueprint format is ONNX.

CODE 

def export_block(block: str, cfg: dict, input_shape, out_path: Path,
                 opset: int = 17) -> dict:
    model = build_block(block, cfg).eval()
    dummy = torch.randn(*input_shape)

    out_path.parent.mkdir(parents=True, exist_ok=True)
    torch.onnx.export(
        model, dummy, out_path.as_posix(),
        input_names=["input"], output_names=["output"],
        opset_version=opset, do_constant_folding=True,
        dynamic_axes=None,  # Static shapes — matches how NAS queries the LUT.
    )
    return {"params": count_params(model), "onnx_path": str(out_path)}
PLAIN ENGLISH

Build the PyTorch module from the block registry and set it to evaluation mode (no training behavior).

Create a random tensor with the right shape — the model never sees this data; it just needs the shape to trace the computation graph.

Export the computation graph to ONNX format — a universal format that TensorRT can read.

Use constant folding and static shapes (no dynamic axes) — matching the exact deployment scenario for accurate latency numbers.

Return the parameter count and the path to the ONNX file.

Source: lut/export/to_onnx.py lines 9–21

The Shape-Shifter

How one mega-network contains thousands of specialized sub-networks

One Network to Rule Them All

Training a neural network from scratch takes days and costs hundreds of dollars in GPU time. What if you could train one network and then pull out thousands of specialized sub-networks for free?

That's exactly what OFA (Once-for-All) does. Think of it like a walk-in closet where every shirt, jacket, and pair of pants was designed to go with every other piece.

1
Train one massive supernet

A single network that supports every possible combination of depth, width, and kernel size

2
Pick a specific configuration

Choose the depth, kernel, and expand ratio for each layer — like selecting an outfit from the closet

3
Extract a concrete sub-network

Pull out a standalone model with the selected settings — instant, O(1) operation

4
It already has pretrained weights!

The sub-network inherits weights that were co-trained to work — within 1-2% of training from scratch

💡
Weight Sharing

This technique is called weight sharing. Instead of training 10,000 separate networks, you train one that contains all of them. MIT's OFA paper uses a technique called progressive shrinking to make this work.

What Can You Vary?

The OFA supernet is elastic along three dimensions. Each one controls a different aspect of the network's shape:

🔑

Kernel Size (ks)

[3, 5, 7] — How wide the filter window is. Bigger kernels capture more spatial context but cost more computation.

📈

Expand Ratio (e)

[3, 4, 6] — How wide the internal channel expansion is. Higher ratios give more capacity but more computation.

📏

Depth (d)

[2, 3, 4] per stage — How many blocks in each of the 5 stages. Deeper = more layers of processing.

🔢
The Search Space is Enormous

With 5 stages, 3 kernel choices, 3 expand choices, and 3 depth choices per stage, there are roughly 2 × 1019 possible architectures. The supernet covers all of them with shared weights.

How Sampling Works

Here's the actual code that extracts a subnet from the supernet. It's surprisingly simple — just 3 lines of real work:

CODE · supernet/sampler.py 

def sample(
    arch_dict: dict,
    supernet: OFAMobileNetV3 | None = None,
) -> nn.Module:
    """Materialise a subnet from {"ks": [...], "e": [...], "d": [...]}.

    Returns a freshly deep-copied MobileNetV3 that owns its weights;
    the caller may train it without touching the supernet's parameters.
    """
    sn = _get_or_load_supernet(supernet)
    sn.set_active_subnet(ks=arch_dict["ks"], e=arch_dict["e"], d=arch_dict["d"])
    return sn.get_active_subnet(preserve_weight=True)
PLAIN ENGLISH

Take an architecture description — a dictionary with lists of kernel sizes, expand ratios, and depths

Optionally accept a pre-loaded supernet (or load one automatically)

Return a PyTorch neural network module

Returns a deep-copied standalone network that owns its own weights — safe to train without affecting the supernet

 

Get the supernet (loading it from the cached 31 MB checkpoint if needed — this load happens once)

Tell the supernet which configuration to activate: set the kernel, expand ratio, and depth for each block

Extract the active sub-network as a new, independent model with copied weights

💡
Not All Slots Are Active

The ks and e arrays always have 20 entries (5 stages × 4 max blocks each), but only the first d[s] entries per stage are actually used. The rest are placeholders. With d=[3,4,2,2,4], only 1 + 3+4+2+2+4 = 16 blocks are active.

Impedance Mismatch

When two systems that need to work together use different data formats, engineers call it an impedance mismatch. The solution is always an adapter that converts one format to the other.

The Shared Topology Table

Both the LUT grid and the translator need to agree on the OFA network's fixed structure. That structure lives in one shared file — the single source of truth:

CODE · catalog/ofa_mbv3.py 

STAGES = [
    {"out_c": 24,  "stride": 2, "se": False, "res_in": 112},
    {"out_c": 40,  "stride": 2, "se": True,  "res_in": 56},
    {"out_c": 80,  "stride": 2, "se": False, "res_in": 28},
    {"out_c": 112, "stride": 1, "se": True,  "res_in": 14},
    {"out_c": 160, "stride": 2, "se": True,  "res_in": 14},
]
PLAIN ENGLISH

The five searchable stages of the OFA-MobileNetV3 backbone:

Stage 0: Output 24 channels, stride 2 (halves resolution: 112→56), no SE attention

Stage 1: Output 40 channels, stride 2 (56→28), with SE attention

Stage 2: Output 80 channels, stride 2 (28→14), no SE

Stage 3: Output 112 channels, stride 1 (stays at 14×14), with SE

Stage 4: Output 160 channels, stride 2 (14→7), with SE

 

📁
Why catalog/, not search/?

This table lives in catalog/ofa_mbv3.py because two consumers need it: the LUT grid generator and the translator. Putting it in the shared catalog layer guarantees they can never drift apart.

How the Walk Works

The translator walks through the topology and emits one LUT-keyed tuple per block:

CODE · search/arch_to_blocks.py 

def arch_to_blocks(arch_dict: dict) -> list[Block]:
    ks, e, d = arch_dict["ks"], arch_dict["e"], arch_dict["d"]
    blocks: list[Block] = []

    def emit(cfg: dict) -> None:
        blocks.append(("mbconv", cfg, input_shape_for("mbconv", cfg)))

    emit(dict(FIRST_BLOCK))  # fixed, non-elastic

    for s, stage in enumerate(STAGES):
        out_c, se = stage["out_c"], stage["se"]
        res_in = stage["res_in"]
        res_out = res_in // stage["stride"]
        for j in range(d[s]):
            slot = MAX_DEPTH * s + j
            if j == 0:
                emit({"in_c": stage_in_c(s), "out_c": out_c, "kernel": ks[slot],
                      "stride": stage["stride"], "expand": e[slot], "se": se,
                      "res": res_in})
            else:
                emit({"in_c": out_c, "out_c": out_c, "kernel": ks[slot],
                      "stride": 1, "expand": e[slot], "se": se, "res": res_out})
    return blocks
PLAIN ENGLISH

Take the arch dict and unpack its three arrays: kernel sizes, expand ratios, per-stage depths

Start with an empty list of blocks

 

Helper: given a block config, compute its input shape and add the (block, cfg, shape) tuple to the list

 

 

First, emit the fixed first block (16→16, k=3, stride=1) — this one never changes

 

For each of the 5 stages...

Read this stage's output channels and SE flag

Read the input resolution

Compute output resolution: input ÷ stride

For each active block in the stage (d[s] blocks)...

Calculate the slot index: stage × 4 + block

Entry block (j=0): prev stage → this stage, at stage stride

Uses previous stage's output channels as input, this stage's output as target

Reads kernel and expand from the arch dict at this slot position

Repeat blocks (j>0): same channels in and out, stride 1

Uses the stage's own output channels for both input and output

Operates at the reduced resolution (after stride was applied by the entry block)

Return the complete ordered list of blocks

The Planned Search Loop

With 2 × 1019 possible architectures, you can't try them all. Like a chess grandmaster who doesn't evaluate every possible move, the search builds intuition and explores smartly.

1
🔍 BO Proposes

Bayesian Optimization suggests a new architecture based on what it has learned so far

2
🧠 Supernet Extracts

3

4
♻️ Net2Net Adapts

If the new arch is close to a previous one, reuse and transform weights

5

Short 5-epoch fine-tune on the target task to get accuracy

6

J(α) goes back to BO, which updates its model and proposes the next candidate

Bayesian Optimization: The Smart Search

Instead of guessing randomly, BO builds an internal model of what works and uses it to decide what to try next. It typically finds better architectures in 30-100 tries than random search finds in 1,000.

🧠

The Surrogate

BO builds an internal model (a Gaussian Process) that predicts how good each architecture will be. Like a grandmaster's intuition about board positions.

🎯

Acquisition

The acquisition function finds the architecture with the best expected improvement. It balances exploring unknowns vs. exploiting what's already known to be good.

📈

Learning

After each evaluation, the surrogate gets smarter. It uses a Hamming kernel to understand which architectures are similar.

💡
The Objective Function

Every candidate is scored by: J(α) = accuracy(α) − λ · latency_ms(α) − μ · max(0, mem(α) − budget)²
Translation: Score = how accurate it is, MINUS a penalty for being slow, MINUS a big penalty if it uses too much memory.

Net2Net: Recycle, Don't Retrain

Net2Net provides two function-preserving operators that let you grow a network without losing what it already learned:

↔️

Net2WiderNet

Make a layer wider (more channels) while keeping exactly the same function. Copies neurons and splits them so the output is identical despite the layer being larger.

↕️

Net2DeeperNet

Insert a new layer initialized to identity (passes input through unchanged). The network is deeper but functionally identical — then training specializes the new layer.

♻️
The Architecture Cache

When BO proposes architecture α_new that differs from a recent α_prev only by a widen or deepen step, Net2Net applies the transform instead of re-extracting from OFA. When the difference is something Net2Net can't handle (kernel change, SE toggle), it falls back to extracting fresh from the supernet. This cuts fine-tuning cost by 2× or more.

Why Not Run PyTorch on the Jetson?

PyTorch needs ~2 GB of RAM just to load. On a device with 8 GB total (shared between CPU and GPU), that would eat into the very memory you're trying to measure. ONNX files are lightweight — the Jetson only needs TensorRT.

The Docker Container

The Jetson runs benchmarks inside a Docker container called lut-runner. Here's why that matters:

🔒

Isolation

Each benchmark runs in a fresh container. No state leaks between runs — measurements are clean and reproducible.

📦

Dependency Pinning

The container pins TensorRT 8.6.2 + CUDA 12.2 + PyCUDA. Host OS updates won't change measurement results.

🚀

GPU Passthrough

The --runtime nvidia flag gives the container direct access to the GPU. No performance penalty from containerization.

CUDA Event Timing

Python's time.perf_counter() adds ~100µs of jitter. On operations that take 0.05ms, that's a 200% error. Instead, the bench uses CUDA events for timing:

CODE · lut/bench/run_bench.py 

samples_ms = []
start_evt = cuda.Event()
end_evt = cuda.Event()
for _ in range(iters):
    start_evt.record(stream)
    ctx.execute_async_v3(stream.handle)
    end_evt.record(stream)
    end_evt.synchronize()
    samples_ms.append(end_evt.time_since(start_evt))
PLAIN ENGLISH

Create an empty list to collect timing samples

Create two GPU-side timestamp markers (like start/stop on a stopwatch)

 

Run the inference 200 times...

Record the GPU timestamp BEFORE running inference

Run one inference on the TensorRT engine (asynchronously on the GPU)

Record the GPU timestamp AFTER the inference

Wait for the GPU to finish (so the timestamps are final)

Calculate elapsed time in milliseconds and save it

💡
Why 50 Warmup Iterations?

The first few GPU runs are slow — the kernel auto-tuner is still picking the fastest algorithm, caches are cold, and clock frequencies haven't ramped up. After ~50 runs, the GPU reaches its steady state and measurements become stable.

The Roofline Model (for Offline Dev)

When you don't have a Jetson handy, the dummy LUT generator uses a roofline model to synthesize realistic latency numbers:

CODE · lut/orchestrate/gen_dummy_lut.py 

def roofline_latency_ms(flops: int, io_bytes: int,
                        rng: random.Random) -> float:
    compute_s = flops / (PEAK_FLOPS * EFFICIENCY)
    mem_s = io_bytes / (PEAK_BW * EFFICIENCY)
    base_ms = max(compute_s, mem_s) * 1000.0 + OVERHEAD_MS
    jittered = base_ms * (1.0 + rng.gauss(0.0, 0.02))
    return max(jittered, OVERHEAD_MS)
PLAIN ENGLISH

Given the number of math operations and bytes to move...

 

How long would the math take? FLOPs ÷ (peak compute × 30% efficiency)

How long would the memory transfers take? Bytes ÷ (peak bandwidth × 30%)

The bottleneck is whichever is slower, plus a 0.02ms kernel launch overhead

Add ±2% random jitter to simulate real measurement noise

Never return less than the overhead floor (even a skip costs something)

⚡

Compute Bound

Large convolutions with many FLOPs. The GPU spends most of its time doing math. Example: 7×7 conv at 112×112.

🚌

Memory Bound

Depthwise convs and 1×1 convs that read lots of data but do little math. The GPU waits on memory. This is most neural net ops!

Anatomy of a LUT Row

Every row in data/lut.jsonl tells a complete story about one block's performance. Here's what each field means:

row_key 16-char SHA-1 hash that uniquely identifies this (block, config, shape) combo
block Block type name (e.g., "mbconv", "conv3x3", "dwconv")
cfg Full configuration dict — channels, kernel, stride, expand ratio, SE flag, resolution
latency_ms Timing stats from 200 iterations — mean, std, p50 (median), p95 (worst-case)
peak_mem_mib GPU memory consumed during inference — important for fitting in the Jetson's 8 GB budget
flops Static FLOPs count — useful for analysis but NOT used for latency prediction (real measurements are better)
achieved_bw_gbps How much bandwidth this block actually uses — helps identify memory-bound vs. compute-bound blocks
